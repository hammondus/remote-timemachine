# Remote Time Machine

Instructions on setting up a MacOS Time Machine to operate over a VPN to a Linux server.
Whenever the Macbook has internet, Time Machine will automatically back up to the remote server.

## Overview
These instructions are for my chosen setup, which is:
- Debian 13 on AWS - t4g.nano (2 cpu 512mb ram) is more than enough
- WireGuard VPN
- Samba file sharing with the vfs_fruit module to work with Time Machine.

Macos. WireGuard app will be installed.
- Each Mac gets a stable WireGuard IP.
- Time Machine points to smb://10.x.x.x./<timemachine-host>.
- One share and quota per Mac.
- The configuration file needed on the Mac is generated by a script on the server.
- Time Machine will run automatically at it's normaly hourly schedule from anywhere with internet.

## Debian Setup
These assume a fresh install of Debian 13 on AWS, and that the server will be soley used for Time Machine.

Create a new EC2 instance with:
- Security group allowing SSH and UDP Port 51820 inbound.
- Key pair for SSH access.
- root gp3 drive with the minimum allowed of 8GB is more than enough.
- sc1 drive for the Time Machine data.

Some little changes to make my life easier

### Setup the drive for the backup data
```bash
admin@timemachine:~$ lsblk
NAME         MAJ:MIN RM  SIZE RO TYPE MOUNTPOINTS
nvme0n1      259:0    0    8G  0 disk
├─nvme0n1p1  259:2    0  7.9G  0 part /
└─nvme0n1p15 259:3    0  127M  0 part /boot/efi
nvme1n1      259:1    0  500G  0 disk
```
In this case, `/dev/nvme1n1` is the drive we want to use for the backup data.

Partition the drive and format it as ext4.
the `-m 0` option sets the reserved blocks to 0, so the drive is fully usable.

```bash
sudo parted -s /dev/nvme1n1 mklabel gpt mkpart primary ext4 0% 100%
sudo mkfs.ext4 -m 0 /dev/nvme1n1p1

admin@timemachine:~$ lsblk -f
nvme1n1
└─nvme1n1p1  ext4   1.0         fc44c527-7611-4ed5-8a68-816f4ee2c3d3
```
`/etc/fstab`
```
UUID=fc44c527-7611-4ed5-8a68-816f4ee2c3d3 /timemachine ext4 defaults,nofail 0 1
```

### Configure WireGuard Server
```bash
cd /etc/wireguard
umask 077
wg genkey | tee server_private.key | wg pubkey > server_public.key
```

`/etc/wireguard/wg0.conf`
```ini
[Interface]
Address = 10.10.0.1/24
ListenPort = 51820
PrivateKey = <contents of server_private.key generated by wg genkey>

# --- Peers added below ---
```
Don't need to add peers. The bash script will automate that part.

```bash
#start the wireguard service
sudo systemctl enable --now wg-quick@wg0

# to check on the status of the WireGuard interface
sudo systemctl status wg-quick@wg0
```

`/etc/samba/smb.conf`
```ini
[global]
   server min protocol = SMB2
   server max protocol = SMB3
   bind interfaces only = yes
   interfaces = lo 10.10.0.1/24
   include = /etc/samba/timemachine-shares.conf
```
`bind interfaces only = yes` and `interfaces = wg0 lo` keeps Samba off the public interface entirely, even if the security groups were misconfigured.

```bash
sudo touch /etc/samba/timemachine-shares.conf
```

get `add-timemachine-client` from this repo.
Check the config at the start. These lines need to be set correctly before running the script.
```ini
WG_SERVER_ENDPOINT="timemachine.example.com:51820"  # public DNS/IP:port
TM_BASE_DIR="/timemachine"
TM_MAX_SIZE="1T"
```

Run the `add-timemachine-client` script to add a new client.
```bash
sudo ./add-timemachine-client <client-name>
```

If it works, you should get the following,
Record the Samba password as this is the only time it will be displayed.

    it is not stored anywhere else in plaintext)

This script will add a new user `tm_Craigs-Macbook` with a generated password.

Add to `/etc/samba/timemachine-shares.conf` to allow Samba access to the share.
```ini
path = /timemachine/Craigs-Macbook
valid users = tm_Craigs-Macbook
read only = no
vfs objects = catia fruit streams_xattr
fruit:time machine = yes
fruit:time machine max size = 1T
```

The `vfs objects = catia fruit streams_xattr` is important for Time Machine compatibility.
`catia` - Translates mac-specific illegal characters like : ? or * to valid characters for Samba.
`fruit` - Enables advaned SMB2/3 protocols that macos expects.
`streams_xattr` - Allows Samba to store extended attributes (metadata) for Time Machine compatibility on the filesystem rather in separate hidden files.


Appended to `/etc/wireguard/wg0.conf` to allow WireGuard access to the share.
```ini
# --- Peers added below ---
[Peer]
# Craigs-Macbook
PublicKey = PQVs-G00Sw53yr84g23X78xIxCPWE=
AllowedIPs = 10.10.0.2/32
```

Created 3 files under `/etc/wireguard/clients/Craigs-Macbook/`
`private.key`
`public.key`
`Craigs-Macbook.conf`

The .conf file is what you need when setting up the WireGuard client on the Mac.
`/etc/wireguard/clients/Craigs-Macbook/Craigs-Macbook.conf`
```ini
[Interface]
PrivateKey = eagfDSsdf0m5BFvfHgvL+r8AEJQuBSAE=
Address = 10.10.0.2/32
DNS = 1.1.1.1

[Peer]
PublicKey = adSsdSEkg435rn6I4qb7ahMsdf3we#wr3qlC5P6HGE4=
Endpoint = timemachine.example.com:51820
AllowedIPs = 10.10.0.1/32
PersistentKeepalive = 25
```

Set permissions on the Time Machine share
```bash
sudo chown tm_Craigs-Macbook:tm_Craigs-Macbook /timemachine/Craigs-Macbook/
```

# Mac Client Setup
Install WireGuard on the Mac and import the `.conf` file.
You can either save the above .conf as a text file on the mac and import it into WireGuard or 'Add Empty Tunnel', give it a name of TimeMachine and paste in the .conf file

If you select On-Demand, then the connection will be established automatically when Time Machine tries to do a backup.
Save.
Click on Enable On-Demand
If it works, the TimeMachine VPN interface should go Active

## Time Machine
If everything is set up correctly, we should be able to connect Time Machine to the server
Connect to the Samba SMB share
Go to Finder, then `Go` -> `Connect to Server...` and enter `smb://10.10.0.1/Craigs-Macbook`
Put in the credentials from the previous script.

If that connects, you should be able to connect Time Machine to it.
`System Preferences` -> `Time Machine` -> `Backup Destinations` and clicking on `Add Backup Disk...`
The Share you just connected to should be available as a backup destination.

Enter a password that is used to encrypt the backup. This is different to the credentials you used to connect to the Samba share.
If you lose this password, you will not be able to recover the backup.

There are likely some files or folders you don't want backed up. You can exclude them by clicking on `Options...` and selecting `Exclude from Backup`.
